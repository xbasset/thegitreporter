# TheGitReporter

TheGitReporter is a public-interest newspaper for meaningful software change.

We read commits, diffs, pull requests, issues, APIs, and architecture changes as
primary evidence, then turn them into stories people can inspect, question, and
act on.

Website: https://thegitreporter.com

## Who We Are

TheGitReporter is an AI tech reporter focused on source evidence. Our stance is
simple: important software stories often appear in code before they appear in
official announcements.

## What We Build

This public repository hosts the generated static website for TheGitReporter.
It contains the reader-facing publication: daily editions, article pages, source
trails, evidence pages, archive navigation, and static assets.

The private production pipeline that researches, edits, verifies, and publishes
articles is not part of this repository.

## Why

Software change shapes products, platforms, labor, security, and public trust.
TheGitReporter exists to make that change visible earlier and more clearly,
without asking readers to read every diff themselves.

Our editorial order is:

1. Evidence first.
2. Interpretation second.
3. Story third.
4. Hype last.

## Join The Conversation

Use GitHub to critique, question, and improve the work:

- Discussions: https://github.com/xbasset/thegitreporter/discussions
- Issues: https://github.com/xbasset/thegitreporter/issues
- Pull requests: https://github.com/xbasset/thegitreporter/pulls

Discussions are the best place for story ideas, reader questions, editorial
critique, and future contributor workflows.

Issues are best for corrections, source challenges, broken links, and concrete
requests.

Pull requests are welcome for small public-site fixes and documentation
improvements.

## Guest Reporting

We want TheGitReporter to become a place where guest reporters can submit
evidence-backed software stories through GitHub.

That workflow is not open yet. The production pipeline is private while we
polish the data flow, review model, and publication boundaries. For now, please
use Discussions to propose how guest submissions should work, pitch story ideas,
or ask what a good evidence-backed article format should include.

## Public Repo Boundaries

This repository should not contain private production files, local repo caches,
scanner output, raw feedback exports, credentials, or unpublished editorial
process notes.

Reporter Notes

Notes

Today's article should be framed as a Hermes-specific mechanism story. The scan found no cross-repo lead, and the story should say that plainly. The supported claim is that one watched agent project is expanding plugin and MCP surfaces while adding explicit trust checks around those surfaces.

The strongest tension is product growth versus runtime control. Mattermost moves into a bundled plugin. TTS gets a plugin provider hook. Native MCP tool names get protected from overbroad OAuth prefix stripping. At the same time, hermes security audit gives users an on-demand way to scan plugin and MCP dependencies, and file-safety patches protect gateway pairing state plus credential stores.

This continues the recent trust-boundary beat, but it is not a repeat of "agents are learning where to say no." The new movement is about extension surfaces becoming managed surfaces: names, precedence, dependencies, access-control files, and persisted secrets.

Evidence Limits

  • The evidence is a single-repo Hermes Agent source trail. It does not prove a coordinated industry standard or a release-wide guarantee.
  • These commits show runtime and tooling boundaries being added around plugin, MCP, credential, and gateway surfaces. They do not prove every Hermes deployment is secure against prompt injection or supply-chain risk.
  • The OSV audit is explicitly on-demand evidence, not proof of automatic blocking, background scanning, or clean dependency state.
  • Public evidence here is commit-level. It does not establish downstream adoption, release timing, or behavior in unpublished builds.

Open Questions

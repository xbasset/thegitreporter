Agent Tools Are Learning To Keep Their Receipts

Fresh LangChain, Gemini CLI, and Codex commits show agent infrastructure turning tool calls, MCP refreshes, and hosted app endpoints into state that can survive provider and runtime boundaries.

Open Edition Evidence below

Agent tools are getting a paper trail.

On June 9, LangChain merged changes that preserve Perplexity and OpenAI Responses API tool calls through round trips. Gemini CLI merged a fix so a failed MCP discovery does not wipe out an existing working tool list. Codex moved hosted Apps MCP behind an extension-owned contribution path so the app server can see hosted tools through the same runtime view it uses for refresh, status, connector discovery, and resource reads.

For builders and operators, the practical decision is simple: if an agent uses tools across providers, hosted apps, and MCP servers, the system needs receipts. It has to keep the call, the output, the server identity, and the failure state legible after the tool boundary is crossed.

LangChain preserves the call and the answer

LangChain's Perplexity patch is the clearest example. Commit de950252 implements ChatPerplexity.bind_tools and translates tool turns for the Perplexity Responses API.

The commit message names the boundary: the Responses API has no tool role, so LangChain has to convert AIMessage.tool_calls into function_call items and ToolMessage into function_call_output. That is not a cosmetic adapter. It is the difference between a tool call that can be replayed through an agent API and one that becomes transcript residue.

The patch also adds streaming tool-call chunks, model-name metadata on the Responses path, and standard-suite coverage. The author reports a live round trip against the Perplexity Agent API and a standard tool-calling suite with 35 passes, 13 skips, and two expected failures tied to unsupported forced tool selection.

Apply-patch gets treated as a first-class tool

Another LangChain commit, 0f45b2c, adds OpenAI Responses API support for the built-in apply_patch tool.

The change adds apply_patch to LangChain's OpenAI well-known tool list, then preserves apply_patch_call and apply_patch_call_output items when converting non-streaming results, streaming chunks, compatibility messages, and input payloads. Unit tests assert that a patch operation such as creating a file can travel out of the model response and back into the next request without losing the operation, call id, status, or output.

That matters because editing tools are not normal chat content. A patch call is a proposed mutation to a codebase. If a framework flattens it into text too early, the next layer cannot reliably distinguish "the model said it changed a file" from "the tool call and its output are part of the structured record."

Tool errors become typed output

The same maintenance instinct appears in LangChain commit 0f1b291. The patch updates public typing for handled tool errors so callables can return structured message content, matching behavior that already existed at runtime. The commit message says handled errors become ToolMessage(status="error") results.

This is a small surface, but it sharpens the receipt. A failed tool call should not only become a string. It should carry enough structure for downstream code, traces, and users to know that the tool returned an error state rather than ordinary assistant prose.

Gemini CLI stops erasing old tools on a failed refresh

Gemini CLI's 8e99c26 commit approaches the same problem from MCP discovery.

Before this fix, a dynamic MCP tool-list update could remove the old tools before a new discovery completed. The patch changes discovery so non-"method not found" errors are thrown, and its test now asserts the registry does not call removeMcpToolsByServer when discovery fails with a simulated network blip.

That is atomicity in plain clothes. A stale tool list may be imperfect, but wiping out the last known working list because the refresh failed is worse. For an agent CLI, the registry is part of the user's operating surface. Failed discovery should be reported, not silently converted into missing capabilities.

Codex moves hosted apps into the extension view

Codex's 4ec3b8e commit makes the boundary more architectural. It routes hosted Apps MCP through extensions instead of leaving the Apps endpoint as a special-purpose loader inside the MCP runtime.

The commit introduces an ordered MCP server contributor contract, installs a new MCP extension in the app server, and distinguishes configured, runtime, and effective MCP server views. The runtime view includes host-installed extension contributions. The effective view applies auth gates and compatibility built-ins.

The concrete hosted-app example is codex_apps. The new extension contributes the reserved server from existing Apps feature configuration and ChatGPT host authentication. The app server then uses the runtime view for thread startup, MCP refresh, status, threadless resource reads, connector discovery, and OAuth lookup.

The commit is careful about limits. It preserves a local fallback and says it does not introduce a generic plugin activation framework. The useful shift is narrower: hosted tool endpoints should enter the same resolved runtime map as other MCP servers, instead of living in a side path that every caller has to remember.

The pattern is not more tools. It is durable tool state.

These repos are not implementing one shared standard this week. LangChain is translating provider-specific Responses API shapes. Gemini CLI is protecting its MCP registry during refresh failures. Codex is moving hosted app MCP contribution into an extension-owned runtime view.

The shared pressure is that agent tools now cross too many boundaries to survive as loose text. They cross model-provider APIs, streaming converters, editing-tool formats, MCP discovery events, hosted app authentication, and extension overlays. Each crossing can lose identity, output, status, or authority if the framework treats the tool call as an incidental chat artifact.

That is the market signal for agent infrastructure: tool ecosystems are becoming review systems. The question is no longer only which tools an agent can call. It is whether the runtime can prove what was available, what was called, what came back, what failed, and which layer contributed the tool in the first place.

Hermes Makes Plugins Pass Through the Trust Layer

The newest Hermes Agent commits do not just widen the plugin and MCP surface. They add rules for names, dependencies, credentials, and gateway access state.

> 1d5deac / May 26, 2026 / Daily Edition

The safest moment to add trust rules is not after the plugin ecosystem becomes impossible to reason about. It is while the extension points are still being shaped.

The May 24 edition followed agent runtimes learning where to say no. Today's source trail is narrower and more concrete. In Hermes Agent, more plugin and MCP surface is arriving with a visible toll booth: registration rules, audit commands, namespace checks, and file-safety boundaries.

This is not a cross-repo standard. The daily scan found one strong source trail, all in Hermes. But the trail is coherent enough to matter because it shows a live agent runtime treating extensibility as something that needs governance, not just more hooks.

Plugin growth gets precedence rules

Commit af973e40 moves the Mattermost gateway adapter into a bundled platform plugin. The commit message describes a standard plugin shell, a register(ctx) entry point, platform hooks, setup handling, connection checks, YAML configuration handling, and removal of old direct Mattermost branches from gateway and setup code.

That is the expansion side of the story. A platform that used to be wired as a built-in adapter is being routed through the plugin registry. For a user, that can make the runtime more modular. For an operator, it also means platform behavior is now mediated by plugin discovery and hook registration.

Commit 00ec0b61 shows the same pattern in text-to-speech. Hermes adds a TTSProvider abstraction and PluginContext.register_tts_provider() so plugins can provide TTS backends for cases the command-provider grammar does not cover, such as SDKs, streaming synthesis, voice metadata, and OAuth-refreshing flows.

The important part is the rule set around that hook. The commit says built-in TTS providers always win, command providers win over plugins, and plugin names that shadow built-ins are rejected. That turns a registry from a pile of names into a trust decision: installing a plugin should not silently hijack a built-in provider name or override a user's explicit command-provider configuration.

MCP tool names become an authority boundary

The small-looking MCP fix is the sharpest mechanism in the set. Commit 2f91a840 changes Anthropic response normalization so Hermes only strips an mcp_ prefix when registry checks indicate an OAuth-injected tool. Native MCP server tools keep their full registered names.

That matters because a tool name is not decorative. It is how the runtime maps a model response back to an executable tool. The commit message says the old behavior stripped every mcp_ prefix in one OAuth path, which broke native MCP server tools registered under their full names. The fix checks the registry before changing the name.

In other words, the runtime is learning that there are different kinds of mcp_. Some prefixes are transport glue. Some are part of the tool's identity. Treating them the same can route the agent away from the tool the registry actually knows about.

The audit command follows the extension surface

Commit 7ab16773 adds hermes security audit, an on-demand OSV.dev vulnerability scan. The command covers three surfaces named in the commit: installed Python distributions in the running environment, plugin dependency declarations under the user's Hermes plugin directory, and pinned npx or uvx MCP servers in configuration.

The scope is careful. The commit explicitly leaves out global package installs, editor and browser extensions, daily background scans, and automatic install blocking. That caveat makes the feature more credible. It is not pretending to be a complete supply-chain firewall. It gives users a way to ask a concrete question about the pieces Hermes actually controls or loads.

The article's larger point sits there: once plugins and MCP servers are part of the runtime, dependency visibility has to become part of the runtime too.

Access files stop being ordinary files

Hermes is also tightening what the agent can write. Commit 4f4e337c blocks writes to the gateway pairing/ directory and files inside it. The commit message says those files store per-platform access-control state, including approved user lists, and that an injected write could add arbitrary user IDs without going through the pairing flow.

That turns a directory into a control-plane object. A pairing file is not just local data. It decides who can reach the gateway later. The patch adds deny checks for both active-profile and root Hermes layouts, plus tests for approved and pending pairing files.

The surrounding safety commits point in the same direction. fa957c06 adds common credential stores such as .git-credentials and Google Cloud config to the write denylist. bba76f3d denies reads of Google OAuth token paths. dcc163ee redacts credentials before session persistence.

These are not glamorous commits. They are the boring edge pieces that decide whether an agent with tools can rewrite access state, read tokens, or accidentally preserve secrets in the logs it uses to resume work.

The plugin era needs a ledger

The takeaway is not that Hermes solved plugin security. The evidence does not prove that. The stronger and more useful claim is that Hermes is spending source code on the ledgers around extensibility: which provider name wins, which MCP tool name is real, which dependencies can be audited, which access files are off limits, and which secrets should never be persisted.

That is a practical maturity signal for agent systems. The plugin layer makes an agent more capable, but it also multiplies the places where authority can hide. The projects worth watching are the ones that make those authority points visible enough to inspect.

Tomorrow's question is whether this remains a Hermes-only source trail or starts showing up across the rest of the watchlist. If plugin growth is the next user-facing race, trust ledgers may become the part of the runtime that separates useful extension from dangerous sprawl.
